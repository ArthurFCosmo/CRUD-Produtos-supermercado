### O que são Requisições HTTP

Pensa assim: você está num restaurante.

- **Você** é o cliente (o frontend, ou um app como Postman)
- **O garçom** é o HTTP
- **A cozinha** é o seu servidor (sua API Spring)

Quando você quer algo, você **faz um pedido ao garçom**. Esse pedido tem uma estrutura:

**"O quê você quer fazer"** — isso são os **verbos HTTP**:

|Verbo|Analogia|O que faz|
|---|---|---|
|`GET`|"Me traz o cardápio"|Busca dados|
|`POST`|"Quero pedir uma pizza"|Cria algo novo|
|`PUT`|"Troca minha pizza por uma lasanha"|Atualiza algo|
|`DELETE`|"Cancela meu pedido"|Deleta algo|

**"Onde você quer"** — isso é a **URL**:

```
GET  /produtos        → lista todos os produtos
GET  /produtos/5      → busca o produto de id 5
POST /produtos        → cria um produto novo
DELETE /produtos/5    → deleta o produto de id 5
```

**"O que você está mandando"** — isso é o **body** (corpo da requisição). Nem todo pedido tem body — um `GET` só pergunta, não manda nada. Mas um `POST` manda os dados do que quer criar:

json

```json
{
  "name": "Arroz",
  "price": 799
}
```

**A cozinha responde** — isso é o **Response**, e vem com um **status code**:

|Código|Significado|
|---|---|
|`200 OK`|Deu certo|
|`201 Created`|Criou com sucesso|
|`404 Not Found`|Não achou o recurso|
|`500 Internal Server Error`|A cozinha pegou fogo|

---

No seu CRUD, cada operação vai ser exatamente um desses verbos batendo numa URL. Quando você criar o `Controller` lá no Spring, é justamente isso que você vai mapear — qual verbo + qual URL chama qual método do seu código.
